# 仕様確定後のモック作成ガイド

## 目的

本実装の前に、利用者が本番相当のGame Fields標準UIで、画面、操作順、情報の見え方を確認できる状態を作ります。モックは完成品のふりをせず、動く部分と未実装部分を明確に分けます。

## 開始条件

利用者が一括質問票へ回答したら、次をAIが整理してモックを作り始めます。

- `GAME_SPEC.md`の必須項目が埋まっている。
- 勝利・終了条件、主な操作、秘密情報、時間切れ、切断・復帰が決まっている。
- 未定・空欄は、AIが安全で一般的な初期値を仮置きし、`GAME_SPEC.md`でAI判断と分かるようにする。

仮置きで進められる事項を一問ずつ聞き返しません。安全性や根本構造に関わり仮置き不能な事項だけ、追加質問を一度にまとめて行います。

## 作るもの

`mock/`には次のGame Fields標準プレビューがプレインストールされています。

- `index.html`: ブラウザで直接開ける広場・入室・部屋ロビー・ゲーム画面
- `styles.css`: 本番相当の色、カード、トップバー、PC・スマホ配置
- `mock.js`: 広場からゲームを選ぶ導線と共通デバッグ操作

外部CDN、API、ログイン、DBへ依存させません。架空のプレイヤーとデータだけを使います。AIはこれらを白紙から作り直さず、`data-game-title`、`data-game-summary`と`#game-slot`を中心にゲーム固有内容だけを編集します。

## 作り方

1. `GAME_SPEC.md`からゲームカード情報、ゲーム固有画面、利用者の操作順を取り出す。
2. `SDK_MODULE_CATALOG.md`から利用する公式モジュールを選び、`APP_REQUIREMENTS.md`の各要件を「該当」「非該当」「本実装時」のいずれかに分類する。
3. 最短の主要フローを操作できるようにする。
4. プレインストール済みのGame Fields標準広場で新作カードを選び、入室前画面、部屋ロビー、ゲーム、同じ部屋への復帰までを確認できるようにする。共通トップバー、カード、参加者欄、ルール、デバッグパネルをゲームごとに再生成しない。
5. 権限のあるホストだけに見える「デバッグ」メニューを用意する。
6. デバッグメニューから、ダミー参加者追加、閲覧プレイヤー視点切替、主要フェーズ切替、待機、時間切れ、切断、入力エラー、結果を再現できるようにする。
7. ダミー参加者を自動進行または一括入力でき、1人で開始から結果まで止まらず確認できるようにする。
8. 進行中断で同じ参加者のロビーへ戻れるようにする。
9. デバッグ権限なしへ切り替え、デバッグUIが消えることと秘密情報が通常視点へ漏れないことを確認できるようにする。
10. ホスト／一般参加者など役割を切り替え、見える操作の差を確認できるようにする。
11. 390px前後のスマホ幅と1280px前後のPC幅で確認する。
12. `MOCK_REVIEW.md`を完成させ、`npm run check:mock`を実行する。

## 利用者への説明

モック作成後は長い技術説明をせず、次を伝えます。

1. 作った画面と試せる操作
2. 仕様のどこを画面へ反映したか
3. 共通要件への主な対応
4. モックのため実際には動かないもの
5. 「実際に画面を見て、変えたいところはありますか？ 特になければ『これでOK』と答えてください」

利用者が本実装を明確に承認するまで、`src/`と`tests/`を個別仕様へ書き換えません。
