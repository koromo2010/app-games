# Game Fields プラットフォーム構想

更新日: 2026-07-21

この文書は、Game Fields を単発ゲーム集ではなく、AI時代の言葉・推理・協力型パーティーゲーム基盤として育てるための長期構想を記録する。

現時点ですべてを実装するという意味ではない。現在の開発判断が将来の展開を妨げないよう、方向性と境界だけを共有するための資料である。

## 1. 基本ビジョン

Game Fields は Steam のような総合ゲーム販売ストアを目指さない。

目指すのは、次の特徴を持つ「友人や他の参加者と集まって遊ぶ知的パーティーゲームの場」である。

- ブラウザからすぐ遊べる
- 言葉、推理、ブラフ、協力、創作を中心とする
- 複数ゲーム間でアカウント、フレンド、実績、戦績を共有する
- LLMをゲーム体験へ組み込みやすい
- ゲームを増やすほど共通DBとプレイデータの価値が高まる

Steamが「ゲームを探して買う場所」なら、Game Fields は「継続して集まり、複数のゲームを遊ぶ場所」と位置付ける。

## 2. 独自性を失わないための範囲

規模が拡大しても、何でも扱う総合プラットフォームにはしない。

主な対象は以下とする。

- 言葉遊び
- 推理
- ブラフ
- 協力
- 即興・創作
- AIとの対話や推論をゲーム性に変換できるもの

FPS、レース、巨大MMORPGなど、制作費競争になりやすい分野へ正面から広げない。Game Fieldsらしい体験を保つことを優先する。

また、将来的に他社・他作者のゲームを扱う場合も、単なるストア化は避ける。共通UI、共通ロビー、共通アカウント、共通安全基準へ自然に統合できる作品だけを対象とする。

## 3. マルチプラットフォーム構想

Game Fields Account を中心に、複数の入口を同一サービスへ接続する。

```text
                 Game Fields Account
                         |
          +--------------+--------------+
          |              |              |
         Web           Steam        iOS / Android
          |              |              |
          +--------------+--------------+
                         |
                    共通サーバー
```

共通化する候補:

- プロフィール
- フレンド
- 戦績
- レーティング
- 実績
- プレイ履歴
- お気に入り
- ゲーム設定
- 購入権・利用権
- モデレーション状態

Steam、Apple、Google等の外部アカウントは、Game Fields Accountに連携する認証手段の一つとして扱う。

これにより、家ではSteam、外ではiPhone、友人の端末ではブラウザという遊び方でも、同一のデータを利用できる状態を目指す。

## 4. Steamとの関係

Steamは競合先ではなく、販売・認知・起動経路として利用する。

想定する役割分担:

- Steam: 購入確認、配布、アップデート、実績、コミュニティ導線
- Game Fields: アカウント、ルーム、ゲーム進行、DB、クロスプレイ、継続運営

Steam版でもGame Fields Accountへログインまたは連携し、Web版と同じサーバーへ接続する構成を基本案とする。

Steam版だけでユーザーを隔離せず、可能なゲームはWeb・Steam・モバイル間でクロスプレイする。過疎をプラットフォーム別に分断しないことを重視する。

## 5. iOS / Androidとの関係

モバイル版も独自アカウントでログインし、共通データを利用する。

ただし、デジタルコンテンツ販売や有料機能は各ストアの課金規約へ適合させる。外部購入権の扱い、アプリ内課金、アカウント削除、Sign in with Apple等は、公開時点の規約に合わせて個別設計する。

## 6. AI利用の基本思想

Game Fieldsは「運営がすべてのAI利用料を負担するサービス」だけを前提にしない。

長期的には、次の複数方式を同じ基盤で扱えるようにする。

1. 利用者自身のAPIキー
2. Game Fieldsが提供する有料枠・クレジット
3. 無料枠や提携プロバイダー
4. ローカルLLM
5. 生成済み・承認済みDBのみを利用する非リアルタイム方式

利用可能なプロバイダー候補:

- OpenAI
- Google Gemini
- Anthropic Claude
- Groq
- OpenRouter
- Ollama等のローカル実行
- 将来の新規プロバイダー

重要なのは、特定のLLM事業者へゲームルールを密結合しないことである。ゲーム側は共通LLMゲートウェイを利用し、モデルや課金元を差し替えられる状態を維持する。

## 7. 顧客側AIリソースという発想

将来的にLLMが一般的な個人インフラになれば、利用者が自分の契約・API・ローカルモデルをゲームへ持ち込む形が自然になる可能性がある。

Game Fieldsは、AIモデルそのものを主商品にするのではなく、各種AIを使って遊べるルール・安全機構・共通基盤を提供する。

```text
利用者のAIリソース
       |
       v
Game Fields 共通LLMゲートウェイ
       |
       v
各ゲームのルール・進行・評価
```

この方式には以下の利点がある。

- 人気が増えるほど運営API費用が爆発する構造を避けられる
- ユーザーがモデルの品質、速度、価格を選べる
- 新モデルが登場してもゲーム本体を作り直さず対応できる
- ローカルLLMを選ぶことでプライバシー重視の遊び方も可能になる

一方、一般利用者にはAPI登録が難しいため、無料体験、設定ガイド、Game Fields提供枠を併用し、持ち込みAIを必須にはしない。

## 8. Steam版と生成AI

Steam版でも外部APIを利用すること自体は可能だが、クライアントへAPIキーを埋め込まない。

リアルタイムAIが不要なゲームでは、以下を基本構成とする。

```text
管理側でLLM生成
       |
自動審査・人間確認
       |
承認済みデータをDB保存
       |
Steam / Web / Mobile がAPI経由で参照
```

たほい屋では、正解説明や読みを事前生成・保存できるため、Steamプレイ中に毎回LLMを呼ぶ必要はない。

SteamクライアントからPostgreSQLへ直接接続せず、認証済みの読み取りAPIを介して承認済み問題だけを返す。

返却候補:

- 問題ID
- 単語
- 読み
- 正解説明
- 難易度
- 言語
- コンテンツ区分
- データセット版

返却しないもの:

- DB接続情報
- APIキー
- 未審査データ
- 管理用評価
- 個人情報
- 生成・編集権限

必要に応じて端末キャッシュを持たせるが、クライアントへ渡した問題データは最終的には解析可能であることを前提にする。

## 9. 単語DBを中核資産とする

LLM自体は多くの開発者が利用できる。Game Fields固有の競争力は、ゲーム向けに整理・評価された単語DBと、その利用実績にある。

中核データの候補:

- 単語と表記揺れ
- 読み
- Zipf値等の客観的頻度
- 語釈・短い正解説明
- 類義語・対義語・関連語
- ワードウルフ用ペア
- ゲーム別採否
- ゲーム別難易度
- 言語
- 安全性・不適切表現フラグ
- 既出履歴
- プレイヤー評価
- 管理者評価
- 生成元、モデル、プロンプト版
- 実際のプレイ結果

概念上の役割:

```text
LLM          = 汎用エンジン
単語DB       = Game Fields固有資産
ゲームルール = 体験設計
プレイ履歴   = 品質改善データ
```

同じ語を複数ゲームで再利用できるため、1回の生成・審査・評価が複数タイトルの品質改善へ効く。

例:

- ワードウルフではペア不採用でも、たほい屋では採用できる
- ワードアウトで使いやすい語が、コードインターセプトや新規ゲームにも使える
- 難易度評価や認知率を複数ゲームで共有できる
- 多言語化時にも、言語ごとのゲーム適性を別管理できる

長期的には、ゲームコードだけでなく「どの語を、どの設定で、どのゲームに出すと面白いか」という選定データが参入障壁になる。

## 10. 現在のDB方針との整合

現在の以下の方針は、この長期構想を支える。

- PostgreSQLを単語、ペア、語釈、評価、既出履歴の正本にする
- Redisはルーム進行、一時状態、制限時間、ロック、キャッシュに限定する
- Zipf値は客観値として固定する
- ゲーム別の採否・難易度・評価を別管理する
- LLM生成結果と生成メタデータを保存する
- Web、Steam、モバイルからDBを直接触らず、用途別APIを介する
- 言語設定とゲームルームを明示的に分離する

## 11. 個人開発としての進め方

最終構想はチーム運営規模になり得るが、現時点で企業規模の仕組みをすべて先行実装しない。

優先順位:

1. Web版のゲーム体験を完成させる
2. 共通モジュールと単語DBを育てる
3. 実際の利用者が回遊する状態を作る
4. 単体でも強いゲームをSteam等へ展開する
5. 利用者と売上の増加に応じて運用・CS・モデレーションを増強する

設計は将来拡張可能にするが、現在の運営とUIは単純に保つ。

## 12. 現時点の判断

- Game Fieldsは独自プラットフォームとして育てる
- Steam、iOS、Androidは同じサービスへの別入口とする
- Steamと正面から総合ストア競争をしない
- 言葉・推理・協力・AIパーティーゲームへ集中する
- 共通アカウント、クロスプレイ、クロスセーブを長期目標に入れる
- LLMプロバイダー非依存を維持する
- 利用者自身のAIリソースを使う方式を重要な選択肢とする
- リアルタイムAIが不要なゲームは承認済みDB参照で動かす
- 単語DB、ゲーム別評価、プレイデータを中核資産として蓄積する

## 13. 実装時の注意

この文書は長期構想であり、現在実装済みの仕様を示すものではない。

新規実装では以下を守る。

- 将来構想を理由に、現在不要な複雑性を持ち込まない
- 認証、課金、ストア規約は実装時点の最新条件を再確認する
- 外部プラットフォーム依存部分はアダプター境界へ閉じ込める
- ゲーム固有ルールを認証・課金・LLMプロバイダーへ直接結合しない
- 単語DBのライセンス、出典、生成履歴、審査状態を追跡可能にする
- クロスプレイ導入時も秘密情報と閲覧範囲をサーバー側で制御する

## 14. SDK制作者別の合言葉付き利用者広場

将来、SDKで作成して運営審査を通過した個人ゲームを、SDK側の制作者別利用者画面から友人等と遊べるようにする。制作者本人が使う編集・module確認・DEBUG画面と、その制作者の利用者が使うプレイ専用画面を同じSDK基盤内で明確に分離する。

- 正規URL候補は `https://play.game-fields.com/<creator-slug>/<locale>` または `https://sdk.game-fields.com/<creator-slug>/play/<locale>` とし、最終hostnameは実装時に決める。`game-fields.com`本体の一般ゲーム一覧へ個人ページRouteを混在させない。
- 1人の制作者に1つの利用者広場を割り当て、その中へ複数の審査済みゲームカードを置く。制作者向け既存URLの編集・DEBUG機能を利用者へ見せない。
- 言語別Roomとコンテンツ可否は本体のアカウント言語ポリシーを維持し、SDK側URLから開始しても認証後のゲーム実行は本体の共通Runtimeへ接続する。
- 制作者別ページは専用の合言葉を必須にできる。合言葉は平文保存せずserver側のhashで照合し、成功後は制作者slugへscopeした短期HttpOnly Cookieを発行する。共通の`PRIVATE_GAME_ACCESS_KEY`を制作者間で使い回さない。
- 合言葉はページとゲームの起動権限だけを与える。Room内の本人確認、actor、戦績、AI利用権限は通常のGame Fieldsアカウントとserver Runtimeを正本とし、合言葉をプレイヤー本人証明に使わない。
- 本体で実行できるのは、運営審査を通り、固定revisionとして`main`へ登録されたゲームだけとする。SDK Previewの未審査HTML、制作者用管理トークン、外部コードを本番ページから直接実行しない。
- 共通Room、時間管理、DEBUG、LLM、単語DB、結果、戦績等は通常ゲームと同じPlatform moduleを利用し、個人ページごとに複製しない。
- 制作者は自分の広場へ表示する採用済みゲーム、説明、並び順、合言葉の更新を申請できるが、`main`、本番DB、Vercel、他制作者の広場への書込み権限は持たない。

これは将来構想であり、SDK側の利用者専用Route、合言葉管理、公開申請UIはまだ実装していない。
