# 開発会話・作業ログ保存ルール

この文書は、ChatGPT等との開発作業で生じた要望、判断経緯、実施結果をGitへ継続保存する運用の正本である。保存先は `docs/DEVELOPMENT_THREAD_LOG.md` とする。

ログは現在仕様の正本ではない。別スレッドや別担当が「なぜその状態になったか」「何が未完了か」を見失わないための参考記録であり、実装時は `docs/README.md` から現行資料とコードを確認する。

## 必須となるタイミング

次のいずれかが発生した作業単位は、利用者から保存依頼がなくても、最終報告前に必ずログを追記する。

- 新しい機能要望、仕様変更、優先順位、命名、数値条件が決まった。
- 設計方針、採用案、見送り、保留、撤回、次回へ持ち越す事項が決まった。
- バグや障害を調査し、原因、影響範囲、未確定事項が分かった。
- コード、設定、DB、Redis、Blob、Vercel、DNS、GitHub等を変更した。
- テスト、ビルド、デプロイ、実機確認を行い、結果が得られた。
- コミット、push、PR、main反映、develop反映を行った。

単なる挨拶、一般知識の質問、既に記録済みの内容の言い換えだけなら追記しなくてよい。

## 記録単位とタイミング

- 1メッセージごとではなく、同じ目的を持つ一連の作業を1項目にまとめる。
- 原則として作業の最終報告前に追記する。
- 長時間または複数段階の作業は、重要な仕様決定、外部設定変更、公開の直後にもチェックポイントを残す。
- コードや文書をコミットする作業では、ログ追記を同じコミットに含める。
- 会話だけで仕様・保留事項が決まった場合も、ログだけを変更してGitへ保存する。
- 保存できない環境では、保存できなかったことと未記録の範囲を最終報告で明示し、次回の最優先事項にする。

## 必須項目

項目はJSTの日付と短い題名を付け、必要な見出しだけを使う。

```markdown
## YYYY-MM-DD — 題名

### 利用者からの要望

- 何を求められたか。

### 判断

- 採用した方針と、将来の担当が必要とする理由。

### 実施結果

- 実装・設定・調査で実際に完了したこと。

### 検証

- 実行したテスト、build、実機確認、デプロイ結果。

### 関連コミット

- `abcdef0` — 要約

### 未対応・保留

- 次に必要な作業、保留理由、未検証範囲。
```

少なくとも、要望または作業目的、確定した判断、現在の結果、未対応・保留を区別する。実装済み、設定済み、自動テスト済み、実機確認済み、本番反映済みを同じ意味で扱わない。

## 保存してはいけないもの

- APIキー、アクセストークン、Cookie、パスワード、接続文字列の値。
- メールアドレス、氏名等の個人情報。
- 合言葉、正解、秘密語、手札、投稿本文等のゲーム秘密情報。
- 内部プロンプト、ツールの生出力、長大なビルドログ、チャット全文の逐語コピー。
- 根拠のない推測、確認していない成功、存在しないコミット番号。

秘密値そのものは書かず、変数名、対象環境、設定済みかどうかだけを記録する。

## 訂正と正本への反映

- 過去ログは原則として書き換えず、誤りや方針変更は新しい項目で訂正する。
- 現行仕様が変わる場合は、ログだけでなく `DEVELOPMENT_HANDOFF.md`、ゲーム別資料、環境変数資料、登録簿、コード等の正本も同じ作業で更新する。
- ログと現行資料が食い違う場合は、現行資料とコードを優先し、差分を確認してから作業する。
- 保留事項を実装済みとして書かない。再開、撤回、完了した時点で新しい項目を追記する。
