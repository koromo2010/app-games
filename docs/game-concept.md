# ゲーム構想メモ

この文書は、`app-games` に実装するゲームの構想・ルール・AI活用方針を整理するための正本です。

- このスレで壁打ちする
- 合意できた内容をこの文書へ反映する
- 実装スレでは、この文書と実際のコードを基準に開発する
- 結論が変わった場合は、古い記述を残さず現在の方針へ更新する

最終更新: 2026-07-12

## 1. プロジェクト全体

### 目的

複数のパーティーゲームを遊べるWebアプリを作る。最初にワードウルフを整備し、その後たほいや等を追加する。

将来的には、ゲーム中に蓄積した評価やプレイ結果を利用するRAGを構築し、AIによるお題生成の品質を継続的に改善する。

### 現在の候補

1. ワードウルフ
2. たほいや
3. その他のオリジナルゲーム

### 著作権上の方針

- 一般的なゲームの仕組みを、独自の文章・画面・データで実装する
- 既存商品の文章、問題集、画像、固有の演出は流用しない
- 辞書を使う場合、語釈文の転載可否と利用条件を確認する
- 特定の現行ボードゲームを実質的に再現する公開版は作らない

## 2. ワードウルフ

### 基本構造

- 市民には共通のお題Aを配る
- 狼には別のお題Bを配る
- プレイヤーは、自分が市民か狼かを知らない
- 狼も仲間が誰かを知らない
- 会話後、全員が「狼だと思う1人」に投票する
- 最多得票者が狼なら市民側の暫定勝利
- 捕まった狼が市民側のお題を当てれば、狼側の逆転勝利
- 最多得票者が市民なら狼側の勝利
- 固有名詞のお題は、片方の名前だけで相方がほぼ一意に推測できる定番ライバル・隣接時代・同一シリーズ・二大ブランドを避ける。同じカテゴリに他の有名候補を4つ以上挙げられる組み合わせを使う

### 同票と決選投票

- 最多得票者が複数の場合、その候補者だけが順番に追加発言する
- 追加発言後、同票候補者だけを対象に決選投票を行う
- 決選投票でも同票なら、追加発言と決選投票を繰り返す
- 最終対象者が1人に決まるまで続ける
- 決選投票の候補者が2人の場合に限り、候補者本人は投票に参加しない。本人同士の票は互いに相手へ入れるだけで結果に意味を持たないため省略する
- 決選投票の候補者が3人以上の場合、候補者本人も投票に参加する。票の振り分けに意味があるため

### 人数と狼数

#### 仮決定

- 4人以下: 狼1人
- 5人: 市民3人・狼2人
- 人数がさらに増えた場合は、設定により狼数を増やせるようにする

#### 5人・狼2人の場合

狼2人には同じお題を配る。狼同士は互いの正体を知らないため、会話が合うことで自分たちを多数派だと誤認しやすくなる。

市民の勝利条件は「各市民の投票先に狼が1人でも含まれる」ではなく、原則として「最多得票者が狼であること」とする。個別投票のどれかが当たれば勝ちにすると、無作為でも狼に当たりやすく、総当たりに近づくため。

### 未決事項

- 狼が複数いる場合、逆転回答を誰が行うか
  - 捕まった狼のみ
  - 狼側で相談して1回答
- 人数ごとの標準的な狼数
- 会話時間
- 投票前後に追加の弁明時間を設けるか
- お題の難易度・カテゴリ設定
- オンライン対戦をどの段階で実装するか

## 3. たほいや

### 基本構想

- 2人以上で開始できる
- 参加者から回答者を1人選ぶ
- 回答者以外の全員に、一般的な大人がまず意味を知らない難語を提示する
- 回答者以外の各プレイヤーが短い偽説明を作る
- 別ルールとして、参加者全員が偽説明を1つずつ作り、全員が1票ずつ投票して最多得票の説明を競う
- どちらのルールでも保存される偽説明は1人1つ、投票は1人1票とし、全員が完了するまでは本人が上書きできる
- 自分の偽説明には投票できない
- 必要な偽説明がすべてそろったら自動で投票へ進み、必要な投票がすべてそろったら自動で採点・結果発表へ進む
- 投稿・投票・時間切れはサーバー側で判定し、ホストの画面が閉じていても自動遷移する。ホストの手動「投票へ進む」「結果へ進む」ボタンも非常用として残す
- 部屋更新は版番号付きで競合時に再適用し、同時投稿の消失と古い画面からのフェーズ巻き戻しを防ぐ
- 共通の制限時間設定を持つ。文字入力は時間切れ時に入力済みの有効な内容を自動採用し、空欄・無効欄だけを時間切れ扱いにして次のフェーズへ進む
- 偽説明作成者は、入力内容の意味と詳しさを保ったまま辞書調の一文へ整える共通LLMボタンを任意で使える
- 偽説明担当に本物の説明を見せるかどうかは、開始前に部屋設定で選ぶ
- 本物の説明を見せる設定は回答者1人ルールだけで使用でき、全員投票ルールでは保存状態を含めて必ずOFFにする
- 本物の説明と偽説明を混ぜ、回答者だけが本物だと思う説明へ投票する
- 本物の説明は意味だけの一文とし、読み方・語源・用例・括弧書きを含めない
- 本物の説明は約10字・20字・30字を中心に、約40字・50字・最大60字も低確率で混在させる。長くなるほど出現確率を下げ、文字数のために意味を無理に引き延ばさない
- 読みと出典・確認情報は結果発表で公開する。開始前設定で「見せる」を選んだ場合は偽説明担当にも公開する
- 回答者1人ルールの投票中は、偽説明担当にも候補一覧と投票進行を表示するが、投票先は結果まで伏せる
- 過去に出題した見出し語はRedisへ記録し、LLM生成・ローカル候補のどちらでも再出題しない
- ワードウルフと同様に、1プレイヤーが保持できるアクティブなたほい屋の部屋は1つだけとし、再ログイン時はその部屋へ復帰する
- 部屋設定はホストだけが変更できるが、現在値は共通の設定表示UIで参加者全員が確認できるようにする
- 結果画面のGood/Bad・理由・自由記述をRAGへ保存し、次回のお題生成に反映する
- LLMのプロバイダー切替は全ゲーム共通ゲートウェイだけで行い、ゲーム側で同じAPIチェーンを再試行しない。お題は部屋とラウンド単位で一時キャッシュする
- たほい屋のお題は品質優先タスクとし、高推論で3候補を生成してから別系統のAIを優先した独立校閲で最良の1候補を選ぶ。通常は生成・校閲の2回に抑え、生成元と校閲元の両方をメタデータへ残す

### 注意事項

- 見出し語そのものと、辞書が作成した語釈文は分けて扱う
- 公開版では、利用条件上問題のないデータを使うか、語釈を独自に作成する
- AI生成した偽語釈・正解語釈について、事実確認と不適切表現の検査方法を設計する

### 未決事項

- 辞書データの出典
- 採点方式
- ワードウルフと共通化するゲームルーム機能

## 4. AIお題生成

### 方針

AIそのものがプレイのたびに自動学習する構造にはしない。アプリ側に経験を保存し、次回生成時に関連する成功例・失敗例を検索してプロンプトへ渡す。

### 固定ルールの例

- 包含関係を避ける（例: 犬 / 動物）
- ほぼ同義で区別できないペアを避ける
- 一部の人しか知らない専門知識に依存しすぎない
- 双方から複数の会話の切り口が生まれる
- 片方だけが極端に具体的にならない
- 最近使用したお題と重複しない

### 構造化出力

AIの出力はJSON Schemaで固定し、少なくとも次を返す。

- 市民のお題
- 狼のお題
- カテゴリ
- 想定難易度
- ペアを採用できる理由
- 想定される会話の切り口
- 注意点

## 5. RAG構想

### 目的

過去のお題ペア、ユーザー評価、プレイ結果、失敗理由を検索し、新しいお題生成へ反映する。

### 基本フロー

1. 人数、狼数、カテゴリ、難易度などの生成条件を受け取る
2. 条件に近い過去データを絞り込む
3. 高評価の成功例と、関連する失敗例を検索する
4. 最近使用したお題を重複禁止リストとして取得する
5. 固定ルールと検索結果をAIへ渡す
6. 新しいお題ペアを生成する
7. プレイ結果と評価を保存する
8. 蓄積データを使って検索・プロンプトを改善する

### 保存候補

#### お題ペア

- ID
- ゲーム種別
- 市民のお題
- 狼のお題
- カテゴリ
- 想定難易度
- 生成日時
- 生成モデル
- プロンプトバージョン
- 採用・却下・停止状態
- Embedding

#### プレイ条件・結果

- お題ペアID
- プレイヤー数
- 狼数
- 勝敗
- 投票結果
- 逆転回答の成否
- プレイ時間

#### 評価

- 楽しかった / 楽しくなかった
- 簡単 / 適切 / 難しい
- 自由記述
- 問題点タグ
  - すぐ狼が分かった
  - 似すぎていた
  - 違いすぎていた
  - 包含関係だった
  - 知識差が大きかった
  - 話題が広がらなかった
  - 不適切な内容だった

### 検索方針

ベクトル検索だけに依存せず、構造化条件と組み合わせる。

1. ゲーム種別、人数、狼数、難易度、カテゴリ等をDB条件で絞る
2. その候補からEmbedding類似度で検索する
3. 成功例・失敗例・使用済み例を異なる役割でプロンプトへ渡す

### 技術候補

現時点の候補であり、実装前に既存構成との整合を確認する。

- Next.js App Router
- PostgreSQL
- pgvector
- OpenAI Responses API
- OpenAI Embeddings
- PrismaまたはDrizzle

### 導入段階

#### Phase 1: 記録可能な基本ゲーム

- ワードウルフのルールを完成
- お題と生成条件を保存可能にする
- モデル名とプロンプトバージョンを記録する

#### Phase 2: 評価収集

- プレイ後の簡易評価を追加
- 問題点タグを追加
- 管理画面で採用・却下できるようにする

#### Phase 3: RAG

- 評価済みデータをEmbedding化
- 成功例と失敗例を検索
- 生成プロンプトへ自動挿入
- 最近のお題を重複回避に使う

#### Phase 4: 評価と最適化

- RAGなし / RAGありを比較
- プロンプトのバージョンごとに成績を比較
- 十分な教師データが蓄積した場合のみ、ファインチューニングを検討

## 6. プロンプト管理

- プロンプトはコード内に散在させず、バージョンを識別できる形で管理する
- 各生成結果にモデルIDとプロンプトバージョンを保存する
- 本番プロンプトをプレイ結果だけで自動更新しない
- 改良候補は既存版と同一の評価セットで比較してから昇格する
- AIモデルを変更した場合も、同一評価セットで回帰確認する
