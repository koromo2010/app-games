# 一人たほい屋（仮）仕様

最終更新: 2026-07-20

## 位置づけ

通常のたほい屋で投稿され、実際に投票選択肢へ出た偽回答を再利用する一人用四択ゲームの準備仕様である。現段階では候補保存・過去データ移行・票集計の基盤だけを実装し、公開ページと出題APIはまだ作らない。

## 問題と候補

- 問題は常に、正解1件と偽回答3件の四択にする。
- 1語あたりの現役偽回答枠は9件とし、出題ごとに3件を選ぶ。
- 9件は保存上限ではない。現役から外れた回答もstatus付きで保持し、物理削除しない。
- 通常のたほい屋で結果まで到達した偽回答は、名前、プレイヤーID、部屋コードを除いて保存する。
- 正解選択肢、時間切れ、デバッグ用ダミー回答、既知のデバッグ定型文は保存候補から除外する。

## 票

- 通常のたほい屋で偽回答が選ばれた回数を `multiplayer_votes` として初期票へ引き継ぐ。
- 一人用で誤答として選ばれた回答には `solo_votes` を1票加える。
- 表示上・候補選別上の合計票は `multiplayer_votes + solo_votes` とするが、分析できるよう内訳は分けて保持する。
- 一人用の同じ出題試行は安定したイベントIDで一度だけ加算する。
- 0票候補とは、合計票が0で、通常たほい屋の有効投票機会または一人用表示機会が1回以上ある候補を指す。

## 現役9件の更新

- 有効候補が10件以上になった時点で現役を9件へ整理する。
- 正解と同義の候補を最優先で除外する。
- 次に0票候補を優先して現役から外す。
- 0票同士では古い候補を先に外し、新しく通常たほい屋で試された候補へ機会を与える。
- 現役から外す処理はstatus変更で行い、回答本文と票イベントは保持する。

## 正解との同義判定

- 回答保存時にはLLMを呼ばず、まず `unreviewed` として保存する。
- その語を一人用へ初めて出す時、未判定候補を正解文とまとめてLLMへ渡す。
- `same` は `excluded_same_as_answer`、判定不能は `review_uncertain`、別の意味は `eligible` として判定結果を保存する。
- 判定済み候補だけで3件揃う場合、以後の出題ではLLMを呼ばない。
- 新候補が増えた時は未判定分だけを追加判定する。
- 正解文のハッシュが変わった場合、旧判定を無効化して再判定する。
- LLM障害時は未判定候補を問題へ出さない。判定済み候補が3件未満なら別の語を選ぶ。

## 保存先と冪等性

- アプリ用Postgresの `tahoiya_decoy_candidates` が匿名候補と集計値の正本。
- `tahoiya_decoy_candidate_events` が通常ラウンド・旧プレイバック・一人用選択の加算イベントを保持する。
- 候補IDは正規化した単語と回答本文から作り、同じ回答を集約する。
- イベントIDはラウンドまたは出題試行から作り、再読込・再移行・複数タブで二重加算しない。
- プレイヤー名、プレイヤーID、メール、部屋コード、投票者一覧を候補DBへ保存しない。

## 過去回答のサルベージ

- 管理画面の単語候補内「一人たほい屋（仮）・回答候補DB」から管理者の再認証後に実行する。
- Redisの `game-replay:v1:*` と、期限内の `tahoiya:room:*` をSCANし、たほい屋の結果だけを対象にする。
- 通常プレイバックは既定30日、お気に入りは期限なし、部屋本体は最終更新から6時間なので、それ以前に消えた回答は復元できない。
- 元のRedisデータは削除しない。サルベージは再実行可能で、既に取り込んだイベントは加算しない。
