# お題候補DB・プレイヤー経験履歴の設計方針

最終更新: 2026-07-18

この文書は、ワードウルフ・たほい屋および今後のお題生成ゲームで、候補DBと「誰が何を見たか」をどう保存するかの確定方針をまとめる。

## 1. 実現したいこと

- 対戦メンバーのうち1人でも経験済みのお題は、その対戦では出さない。
- 保存済み候補の中から、今回の参加者全員が未経験のものを先に再利用する。
- 未経験候補がなければLLMで新規生成し、候補DBへ追加する。
- ユーザー数が増えても、1つのお題レコードへユーザーIDが無制限に増えない構造にする。
- ゲームごとに「経験済み」の単位を変えられる共通設計にする。

これは勝敗履歴ではなく、表示・出題された問題の経験履歴である。勝敗、正解、不正解に関係なく、参加者が問題を見た時点で経験済みとして扱う。

## 2. 現状と問題点

現状の `lib/tahoiya-topic-catalog.ts` は各問題に `experiencedPlayerIds` を持つ。`lib/wordwolf-topic-catalog.ts` も、候補レコード側に経験済みプレイヤーIDを保存する。

この形は初期実装としては単純だが、人気候補ほど1レコード内のID配列が肥大化する。更新のたびに大きなJSONを読み書きし、同一レコードへの更新集中、通信量増加、Redis値サイズ増加を招く。そのため、長期運用の正規形にはしない。

## 3. 確定する責務分離

### 候補DB側

候補そのものと、候補全体の品質・利用統計だけを持つ。

- 安定した候補ID
- 単語またはワードペアと説明
- 難易度、カテゴリ、出典
- 生成元、モデル、RAGメタデータ
- Good/Bad集計
- 全体利用回数、最終利用日時
- 利用停止・再利用禁止フラグ

候補レコードへプレイヤーID配列を持たせない。

### プレイヤー側

プレイヤーごと・ゲームごとに、経験済み候補IDのRedis Setを持つ。

概念上のキーは次の形とする。

```text
game-history:v2:tahoiya:<playerId>
game-history:v3:wordwolf:daily:<YYYYMMDD JST>:<playerId>
game-history:v3:wordwolf:pair:<playerId>
```

Setのmemberは表示文字列そのものではなく、正規化後の内容から作った安定IDを使う。プレイヤー名やメールアドレスは使わず、内部プレイヤーIDだけをキーにする。

## 4. ゲーム別の経験単位

### たほい屋

- 経験単位は見出し語1語。
- 同じ見出し語は、説明や出典が変わっても同じ問題として扱う。
- 正規化した見出し語から候補IDを作る。
- 今回の参加者の誰か1人でもその候補IDを経験済みなら除外する。

### ワードウルフ

- 同日中はペアに関係なく、見た単語をそのプレイヤーへ再出題しない。日次SetはJSTの日付別キーとし、3日TTLで自動削除する。
- 中期の経験禁則は「ワードペア」だけにし、標準30日で解除する。期間は `WORDWOLF_PAIR_COOLDOWN_DAYS` で変更できる。
- 個々の単語を恒久除外しない。同じ単語が別の組み合わせで翌日以降に再登場することは許可する。
- 村側・狼側を入れ替えても同じペアになるよう、2語を正規化して順序非依存のペアIDを作る。
- ペア履歴はプレイヤー別Sorted Setへ保存し、scoreを最終出題Unix時刻とする。参加者の誰かが期間内に見たペアは除外する。
- 同じペアでも、参加者全員が未経験なら別の対戦で再利用できる。
- 期限切れmemberは候補記録時に `ZREMRANGEBYSCORE` で削除する。個別の禁則解除は `ZREM`（実装上は `releaseWordWolfPairCooldown`）で行える。

Bad評価語の除外は、日次単語履歴・30日ペア履歴とは別の品質フィルターとして扱う。

### 今後追加するゲーム

新ゲームは、履歴保存を複製せず共通ストアへ次を渡す。

- `gameId`
- `playerIds`
- `artifactId` または `artifactIds`
- そのゲームの経験単位を作る正規化関数

## 5. 出題時の選択手順

1. ゲーム・難易度・品質条件から保存候補を少数のshortlistへ絞る。
2. shortlistの各候補について、参加者全員の経験Setへmembership checkを行う。
3. 誰のSetにも存在しない候補だけを残す。
4. `Good - Bad`、Good数、Badの少なさ、全体利用回数、最終利用日時の順で優先する。
5. 対象がなければLLMで新規生成する。
6. 採用候補を候補DBへ保存し、参加者全員の経験Setへ候補IDを追加する。

参加者全員のSetを `SMEMBERS` で丸ごとアプリへ読み込まない。shortlistに対する `SMISMEMBER` / `SISMEMBER` をpipeline化し、参加人数×候補数の必要な判定だけを行う。

## 6. 経験済みを記録するタイミング

- 部屋へお題が正式採用され、参加者へそのラウンドが開始された時点で記録する。
- ゲーム終了時まで待たない。途中退出、時間切れ、部屋解散でも、すでに見た問題は経験済みである。
- デバッグの生成プレビューは実戦経験に含めず、プレイヤー履歴へ追加しない。
- 候補の利用回数更新と参加者履歴追加は、可能ならLuaまたは同等の原子的処理へまとめる。

## 7. 移行方針

現在の候補レコードにある `experiencedPlayerIds` はすぐに削除せず、次の順で移行する。

1. 共通のプレイヤー経験履歴ストアを追加する。
2. 既存候補DBを走査する一度きりの移行処理で、各 `experiencedPlayerIds` をプレイヤー別Setへ反転登録する。
3. 移行期間は新旧両方を読めるようにするが、新規記録はプレイヤー側Setを正とする。
4. 件数とサンプル照合後、候補レコードへのプレイヤーID追記を停止する。
5. 十分な互換期間後に `experiencedPlayerIds` を候補JSONから削除する。

移行完了前に旧配列を消すと過去履歴が失われ、既出問題が再出題されるため禁止する。

## 8. 実装時の注意

- 候補IDの正規化仕様は導入後に変更しない。変更する場合はバージョンを上げて移行する。
- Redisキーはゲーム別に分離し、別ゲームの同じ単語を混同しない。
- 1プレイヤー1アクティブ部屋の既存制約は維持する。
- たほい屋の履歴Setは原則TTLなし。ワードウルフだけは上記の商品仕様どおり日次SetにTTLを設定し、ペアSorted Setを30日で掃除する。
- メモリ使用量は「候補数×全ユーザー配列」ではなく「実際に遊ばれた延べ問題数」に比例する。監視時は総Set member数とプレイヤー別上位件数を見る。
- アカウント削除機能を追加するときは、そのプレイヤーのゲーム別履歴キーも削除対象にする。
- 候補選択ロジックと履歴永続化は共通モジュールへ置き、ゲームAPIへRedisコマンドを重複実装しない。

## 9. 現在の実装状態

2026-07-18時点でワードウルフはv3履歴へ移行済み。旧 `wordwolf:topic:word-experience:v1` は破壊せずアーカイブとして残すが、新規の判定・記録には使わない。旧形式は単語別プレイヤー配列で、過去の正確なペアと出題日時を復元できないため、30日ペア履歴は切替日から蓄積する。

たほい屋は共通DBから、秘境では `0 < 実質Zipf < 3`、魔境では `実質Zipf = 0` の語を通常出題の素材として選ぶ。抽出後に既存の共通LLM経路で読みと正解文を生成・独立校閲し、LLMは難易度を再判定しない。完成済み `tahoiya_topics` / `active_tahoiya_topics` は移行・管理用カタログとして維持する。新規の既出履歴は `game-history:v2:tahoiya:<playerId>` へ保存する。履歴memberはNFKC正規化した見出し語のSHA-256から作る `word-v1:` IDで、候補へプレイヤー配列を追記しない。移行中は旧Redis Hash `tahoiya:topic:catalog:v1` も読み取り、旧 `experiencedPlayerIds` と新Setのどちらかで経験済みなら除外する。

旧Redisのお題本体と経験配列は、develop Preview管理画面の「たほい屋DB移行」から冪等に共通DB／プレイヤー別Setへ移す。元Hashは削除しない。新しいLLM審査済み候補は `definition` draftとして共通DBへ送り、管理者が採用した時だけ `tahoiya_topics` とたほい屋用eligibilityへ昇格する。

主な対象ファイル:

- `lib/tahoiya-topic-catalog.ts`
- `lib/wordwolf-topic-catalog.ts`
- `app/api/tahoiya/topic/route.ts`
- `app/api/wordwolf/topic/route.ts`
- `lib/wordwolf-topic-history-store.ts`
